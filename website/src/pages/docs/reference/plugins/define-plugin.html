definePlugin API

The definePlugin function is the entry point for extending Coralite's functionality. It allows you to inject server-side logic, register custom templates, and most importantly, bundle and configure client-side scripts that can be used by your components.

The definePlugin function accepts a configuration object with the following blocks:

import { definePlugin } from 'coralite' const myPlugin = definePlugin({ name: 'my-plugin', // Required: Unique name for the plugin // Optional: Object of Two-Phase Curried functions mapped to explicit context exports: { log: (context) => () => { console.log(context.state, context.page) } }, // Optional: Main server-side logic method: (options, context) => { // ... implementation }, // Optional: Register custom components components: ['./path/to/component.html'], // Optional: Client-side script configuration client: { context: { /* ... */ }, config: { /* ... */ } }, // Optional: Server-side lifecycle hooks onPageSet: async (data) => { /* ... */ }, // ... other hooks })

The exports property is an object of functions that processes content using plugin logic.

These functions use a Two-Phase Currying pattern to receive the CoralitePluginContext explicitly.

export default definePlugin({ name: 'logger-plugin', exports: { log: (context) => (message) => { // The plugin context is passed explicitly console.log(`[${context.module.name}] ${message}`, context.state); // Returning a state patch return { logged: true }; } } })

Plugins can hook into Coralite's build lifecycle to perform actions when pages or components are processed.

  • onPageSet({ elements, state, page, data, app }): Called when a page is created.
  • onPageUpdate({ elements, page, newValue, oldValue, app }): Called when a page is updated.
  • onPageDelete({ ...data, app }): Called when a page is deleted.
  • onComponentSet({ component, app }): Called when a component is registered.
  • onBeforeBuild({ path, options, app }): Called before the build process starts.
  • onAfterBuild({ results, error, duration, app }): Called when the entire build process finishes.

The client block is powerful: it allows you to inject code that runs in the browser. This code is bundled with your application.

Define utilities that will be flattened directly onto the context object in defineComponent scripts in the browser.

definePlugin({ name: 'utils', client: { context: { formatDate: (globalContext) => ({ state }) => (date) => { return new Date(date).toLocaleDateString() } } } })

Note: Context utilities must use the Two-Phase Currying System.

The client.config object allows you to pass static configuration data from the server (where the plugin is defined) to the client. This is useful for API keys, theme settings, or feature flags.

definePlugin({ name: 'analytics', client: { config: { trackingId: 'UA-123456-7', debug: true }, // ... } })

The client.config is automatically passed as the config property on the globalContext passed to your context factories.

// Inside client.context context: { trackEvent: (globalContext) => ({ state }) => async (eventName) => { const { trackingId } = globalContext.config const { default: analyticsLib } = await import('analytics-lib') analyticsLib.send(trackingId, eventName) } }

Let's build a plugin that triggers a confetti explosion using a remote library and allows configuration of particle count.

import { definePlugin } from 'coralite' export default definePlugin({ name: 'confetti-plugin', client: { // Define default configuration config: { particleCount: 100, spread: 70 }, // Create the context utility context: { explode: (globalContext) => ({ state }) => async () => { // Access config from global context const config = globalContext.config // Dynamically import the library (auto-bundled by AST) const { default: confetti } = await import('https://esm.sh/canvas-confetti@1.6.0') // Use the library with the config confetti({ particleCount: config.particleCount, spread: config.spread, origin: { y: 0.6 } }) } } } })
